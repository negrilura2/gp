1. 主线 A：把“深度学习升级”真正做成一个闭环
目的：让 AI 部分从“功能都在”变成“ 一套完整、可讲故事的研究链路 ”。

- A1. 训练策略实验封闭
  
  - 用现有 Trainer + ecapa.yaml ，设计两组实验：
    - 固定 Epoch（比如 60）；
    - EarlyStopping（你现在的配置）；
  - 输出：
    - 训练时间对比（epoch 数、总秒数）；
    - EER / Acc 对比；
  - 结果以表格 + 1–2 张图（Loss/Acc/EER 随 epoch 变化）整理到 reports/strategy/ ，并在 deep_learning_upgrade_plan.md 或论文里写成一节。
- A2. 嵌入可视化标准化
  
  - 完整打通 scripts/analysis/plot_embedding.py ：从 YAML 读取模型路径、特征目录，调用统一的 embedding 提取接口（建议新建 voice_engine/embedding.py 或从 trainer.py 抽一个工具模块出来）；
  - 输出统一命名的 PNG： reports/plots/embedding/tsne_{feature_type}_{model_name}.png ；
  - 在前端 Dashboard 加一块静态图展示（或者静态图+简单解释）。
- A3. Score Normalization 最小落地
  
  - 先做“离线实验版”：
    - 在 scripts/evaluate.py 中增加 --score_norm 参数（none / znorm / snorm），利用你在 trainer.py 里已有的 compute_cohort_stats + apply_score_norm ；
    - 生成对比结果：EER + minDCF 对比表，输出到 reports/score_norm/ ；
  - 再考虑是否把其中一种（比如 S-Norm）接到 VoiceService.verify() 中，并通过配置开关控制，避免影响现有 demo 稳定性。
2. 主线 B：清理关键技术债，防止 Trainer & Scripts 变成真正屎山
- B1. 拆分 trainer 里的“通用评估工具”
  
  - 把下面这些函数移到一个新模块（比如 voice_engine/metrics.py 或 voice_engine/eval_utils.py ）：
    - compute_pairs_scores / compute_eer_from_embeddings / eer_from_roc / compute_mindcf / compute_score_hist / compute_calibration ；
    - extract_all_embeddings / build_templates （如果多处用到）；
  - Trainer 只保留训练相关逻辑，analysis/evaluate 脚本从新模块调用。
  - 这样 Trainer 文件体积下降、职责更清晰，也方便之后再加其他模型时复用。
- B2. 给 VoiceService 增加正式的 reload / set_model API，替换掉 model_loader 的 hack
  
  - 在 VoiceService 内部实现一个类方法：
    
    ```
    @classmethod
    def reload(cls, 
    model_path=None, device=None):
        cls._instance = cls
        (model_path=model_path, 
        device=device)
        return cls._instance
    ```
  - model_loader.set_model_path() 改为调用 VoiceService.reload() ，避免直接操作 _instance 。
  - 这样单例模式更稳，后续如果要做多模型/版本管理也更好扩展。
- B3. 统一 README 与当前脚本/配置入口
  
  - 把 README 中的命令更新为现在实际在用的形式，比如：
    - python -m scripts.data.feature_extraction --config configs/data.yaml
    - python -m scripts.train --config configs/ecapa.yaml
    - python -m scripts.evaluate --config configs/eval.yaml
    - python -m scripts.analysis.noise_robustness --config configs/analysis.yaml
  - 标明“旧路径已废弃，仅保留代码做参考”，避免未来你自己或别人走错路。
3. 主线 C：工程化上选“几拳干净利落的升级”，而不是全都铺开
这个阶段不必一上来搞 Kafka / K8s 全家桶，可以按 工程化与前端优化规划.md 里建议的顺序，选 2–3 个最能在答辩/简历上加分的：

- C1. AI Service 正式独立（最推荐的一个工程化升级）
  
  - 把 VoiceService / 推理逻辑抽到一个独立的 FastAPI 服务（可以沿用当前 voice_engine 包），提供 /health / /embed / /verify ；
  - Django 通过 HTTP 调用，形成“Backend + AI Service”的结构，这个对整个项目定位提升非常明显。
- C2. Docker Compose 打通 DB + Backend + Frontend + AI Service
  
  - 在 docker-compose.yml 里增加 DB 和 AI Service，做到：
    - docker compose up → 整套系统起来；
    - 配置通过 .env.docker 统一注入。
- C3. 前端增加 1–2 个“高价值”可视化组件
  
  - 利用 ECharts 展示 ROC/EER 或 Noise 曲线；
  - 或嵌入一张静态的 t-SNE 聚类图，加上解说文字。
  - 不用追求“酷炫动效”，重点是“ 跟你 Level 1–6 的实验闭环对齐 ”。
